Oligo Security

Security Admin Guide

// May 2026
Purpose

This guide explains how to securely set up, configure, operate, and decommission top-level administrative accounts within the Oligo platform. It is intended for customer administrators responsible for managing their organization's Oligo environment.

A top-level administrative account is defined as the highest level of admin privilege a user within a customer's Oligo environment can hold. This guide does not cover Oligo's internal infrastructure or backend administration.

Top-Level Administrative Account Role Definitions

Oligo enforces role-based access control (RBAC) across its platform. Two user roles are available to customer administrators:

Admin
  • Full access to manage resources and members across the Oligo environment.
  • Can assign and modify roles for other users.
  • Can configure integrations (Slack and Jira) and manage project membership.
  • Can select the Default (all) view to see all resources across the organization without project context.
  • Can create, modify, and remove projects and associated resources.
View-only
  • Read access to resources within assigned projects only.
  • Cannot make changes to settings, integrations, roles, or project membership.
  • Users assigned to two or more projects use the project context selector to switch between project views.
  • Cannot access resources outside of their assigned project membership.

Access is governed by least privilege principles. Users see only what their role and project membership permit, as documented in Oligo's Access Control Policy and Information Security Policy.

Admin Account Lifecycle Procedures

Account Setup
Customer administrators can create new user accounts directly within the Oligo UI or via SSO/SCIM integration, depending on the organization's identity management configuration. During setup, the provisioning administrator assigns the appropriate role (Admin or View-only) in line with the user's responsibilities.
MFA Requirements
Multi-factor authentication is enforced at the platform level via Palantir's FedRAMP-authorized infrastructure, which underpins the Oligo environment. MFA configuration and enforcement is managed through Palantir's Control Panel. Relevant documentation is publicly accessible at:
Authentication and Access Controls: palantir.com/docs/foundry/platform-security-management/overview
Configuration
Once an account is created, the provisioning administrator should:
  • Assign the correct role based on the user's responsibilities.
  • Add the user to relevant projects only — apply least privilege.
  • Confirm integrations are scoped correctly before granting Admin access.
Decommissioning
When a user no longer requires access, the customer administrator follows one of two paths depending on how accounts are managed:
Method Steps
Via SCIM Deactivate the user in the Identity Provider (IDP). The user is automatically deactivated in Oligo once deactivated in the IDP.
Without SCIM Navigate to Settings, click on the user's role, and change to Viewer. This removes admin privileges immediately.
Oligo's Access Control Policy requires that access revocation is completed within 48 hours of a request being raised.

Admin-Controlled Security Settings

The table below describes all security-relevant settings that a customer administrator can configure within Oligo, along with the associated function, security impact, and recommended value.

Setting Admin Action Security Impact Recommended Value
User Role Assignment Assign Admin or View-only role to users Over-privileged users can modify resources, integrations, and member access across all projects Assign Admin only where full management access is required. Default new users to View-only.
Slack Integration Enable/disable Slack notifications and configure connected workspace Misconfigured integration may expose vulnerability alerts to unintended Slack channels or users Connect only to the designated security channel. Review channel membership before enabling.
Jira Integration Enable/disable Jira ticketing and configure connected project Incorrect project mapping may route security findings to non-security teams or expose data outside intended scope Map to a dedicated security project. Restrict Jira project access to relevant team members.
Project Membership Add or remove users from projects; control which resources are included in each project Users with project membership access all resources within that project. Incorrect membership grants unintended data visibility. Apply least privilege. Review project membership on a regular basis.
